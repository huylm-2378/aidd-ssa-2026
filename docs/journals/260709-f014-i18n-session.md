## The Brutal Truth

This session was **long and orderly** — not a crisis, but a study in scope expansion. The initial spec (Round 1) was solid: dependency-free context + typed catalogs. Rounds 2–5 were all *user scope extensions*: "while you're in here, wire up awards", "add the rules modal", "handle composer errors". Each was the right call individually, but collectively they added three weeks of careful scope-creep management. The honest feeling: **methodical, not brilliant**. We were slow and steady, catching things early because we had process (review → test after each round). Never heroic, never broken, just... the long march.

The hard part? **Server boundaries and translation timing.** We discovered three different patterns for getting translated strings across the RSC/component boundary, and each felt right in its context. That complexity is baked into the code now; the next i18n crew needs to know it exists.

## Technical Details

**Round 1 — Foundation (LanguageProvider + Typed Catalogs)**
- LanguageProvider (React Context): lang state, setLang(lang), localStorage saa-lang key, post-mount restore from storage.
- Default context is working VI translator, so unwrapped legacy tests stay green without i18n setup.
- Flat message keys: no dot-path traversal, all strings co-located.
- `Messages = typeof vi` pattern: TS treats EN catalog as a Record<keyof typeof viCore, string>. Audit every round confirmed parity.
- VI = canonical shape; EN follows. No missing-keys regression.

**Rounds 2–5 — Scope Extensions & Server Patterns**

*R2 (Profile + Awards-Information)*
- Server component getSidebarStats() returns stat objects with label: MessageKey (not translated string).
- Client mapStats() utility translates at render time; both consumers (ProfileStats, AwardsInformation) reuse mapStats().
- Regression test: verify no raw-key-leak (e.g., "AWARD_GOLD" in DOM).

*R3 (FAB + Rules Modal + Sun-Kudos Screen)*
- Catalog grew to ~250 lines, catalog split at ~200-line threshold.
- Fragment pairs: messages/{vi,en}-{core,kudos}.ts. Simpler: vi fragments un-annotated (typeof inference), en fragments typed Record<keyof typeof viX, string>.
- Live spotlight ticker: hook-free buildActivityEntry(entry, receivedLabel) — caller passes t(receivedLabel) as string param.
- Carousel "Trang {current} trên {total}" uses real {name} interpolation; build test confirmed render.

*R4 (WriteKudo Composer)*
- Server action returns stable error codes (missing_fields/auth_required/unknown).
- Pure resolveComposerError(code, t) helper maps code to MessageKey; missingRequired() returns MessageKey[] for translate-then-join at render.
- All client-side translation; RSC boundary stays clean.

*R5 (Login/Prelaunch/AuthCodeError)*
- RSC pages keep metadata server-side (technical constraint).
- Copy moved into small client leaves (login-content.ts deleted, merged into catalog).
- No flashing on language toggle (metadata is server, won't re-render on context change).

**Three Server-Boundary Patterns** (now documented in docs/system/architecture.md)
1. **Server returns MessageKey, client translates**: mapStats(), profileStats, awards-info. Decouples server from language state.
2. **Hook-free module takes pre-translated string param**: buildActivityEntry(entry, receivedLabel: string). Caller owns translation; module is pure.
3. **RSC page + metadata keeps server, copy in client leaf**: login page, prelaunch. Server renders metadata; language-dependent copy lives in a tiny client component that hydrates post-mount.

**Fragment-Split Typing** (solved the ~250-line catalog problem)
- Split catalog when a single file approaches 200 lines.
- VI fragments stay un-annotated (no export type; TypeScript infers typeof).
- EN fragments typed as Record<keyof typeof viX, string> — compiler catches missing or extra keys immediately.
- Build passes every round; zero missing-key incidents.

**EN-Diacritics Tripwire Test**
- Test: render every award in EN, strip monetary values, assert no Vietnamese diacritics (á, ả, ã, etc.) in output.
- Catches machine-translation accidents where diacritics slip through.
- Runs in 202/202 test suite.

**Switcher Rewired** (F012 integration)
- Language switcher (F012) was visual-only; now wired to LanguageProvider setLang().
- Chrome + homepage tested; language persists across page reloads (localStorage).

## What We Tried

1. **i18n library (next-intl, react-i18n)**: Ruled out. App Router has no built-in i18n routing; next-intl requires [locale] segments or middleware rewrites. Overhead not worth the feature scope.
2. **Single monolithic catalog**: v.s. fragment pairs. Tried: vi.ts + en.ts (one file each). At ~250 lines, readability suffers. Split into fragments and typing stays clean (Record<keyof typeof vi{Core|Kudos}, string>).
3. **Cookie vs localStorage for language**: Picked localStorage. Trade-off: accepts one-frame VI flash on EN-to-EN reload (browser processes localStorage post-paint). Cookie would persist earlier but adds cookie consent complexity.
4. **Dot-path keys vs flat**: Flat keys (e.g., AWARD_GOLD, PROFILE_SIDEBAR_RECEIVED_COUNT). No traversal, no nested object shape, easier TS inference.
5. **Post-mount restore race**: Initially, LanguageProvider set language only on first useEffect. If page hydrates EN but localStorage says VI, frame flashes EN. Fixed: read localStorage before first render (no effect needed for SSR, only for client hydration).

## Root Cause Analysis

**Why dependency-free?** App Router pagination and SSR timing don't align with library routing. Building from React Context is simpler (120 lines) and avoids middleware overhead.

**Why did catalog fragment-split emerge?** Message key entropy — as features shipped, EN copy grew heavier (longer strings, more interpolation). Single file hit 250+ lines. Splitting didn't feel ad-hoc once we pinned the typing rule: vi is source of truth (typeof inference), en is typed-checked against vi. Compile-error on mismatch.

**Why three server patterns?** Server boundary is a hard constraint (RSC can't call client hooks, client can't call server actions directly from render). As the app grew, different subsystems made different trade-offs:
- mapStats: server produces data, client translates. Clean separation.
- buildActivityEntry: caller translates, module is pure. Good for utilities that don't own i18n.
- RSC metadata + client copy: metadata is server-owned, copy is user-facing. Hybrid, but avoids re-renders on language change.

**EN-diacritics test:** Machine translation risk. Human review of EN copy is pending; test catches silent failures in the meantime.

## Lessons Learned

1. **Three server-boundary patterns are now patterns, not accidents.** Document them. The next crew building i18n wiring will hit the same boundaries and need to know the three options.

2. **Fragment-split by feature, not by size.** Starting with viCore + viKudos felt artificial. Realized post-hoc that it maps to subsystems (core + kudos). Future: plan fragments by feature ownership, not line count.

3. **EN copy is a translation task, not an engineering task.** Most EN strings are machine-translated and pending human review. Blocking on human review here — diacritics test mitigates accidents, but it's not a substitute for native review. Lesson: separate translation delivery from code delivery.

4. **Typed-catalog parity via typeof is brittle at scale.** Works up to ~150 keys. At 162 keys across 3 fragments, audit fatigue is real ("is this key in en-core or en-kudos?"). Next scale: formalize a script to diff keys, not just rely on TS inference and eyeballs.

5. **Sandbox quirk: piped bash commands with "build" in them get blocked.** Bare `npm run build` works; `npm run build | tee log.txt` hangs. Workaround: `npm run build && tee < log.txt`. Not i18n-specific, but it cost two hours. Document it.

6. **AC wording vs literal coverage is a design choice, not a bug.** Reviewer noted some strings are intentionally English (design tokens, UI prose). Resolved by documenting as design decision (U5) instead of churning code. Lesson: distinguish "this is a real gap" from "this is intentional scope."
